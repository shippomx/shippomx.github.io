应用程序释放内存竟然不能减少系统物理内存占用

哪吒藕霸2023-05-112023-09-28

问题背景

某产品线在版本测试中发现，当用被测程序启动若干进程之后，再启动新的进程会因为系统内存不足而失败，而此时已启动运行的进程数量还远少于项目设计的数量。

品线对程序的内存占用进行了分析，结果表明，该程序仅在一个耗时很短的初始化过程中通过malloc多次分配了动态内存，并将其中绝大部分都在初始化完成后通过free释放掉了，后续运行中使用的动态内存并不多；并且，由于多个程序实例是两两间隔一段时间陆续启动的，也不存在多个实例同时进行初始化从而消耗大量内存的场景。

根据该程序的进程对应的maps信息，开发人员发现，程序在完成初始化后，虽然通过free释放了之前分配的大部分内存，但进程的物理内存占用在后续持续运行过程中并没有减少。这是导致过量物理内存占用的直接原因。

那么，程序释放了内存为什么不能减少系统物理内存占用呢？

本文将对这一案例进行分析，并介绍glibc的动态内存管理机制的主要特点和应用建议。

问题分析

我们根据产品线程序在初始化过程中的动态内存使用逻辑，抽象出以下动态内存应用模型：

这样的动态内存分配-释放模式看起来是没有问题的，但在glibc库的动态内存分配器ptmalloc中，用户 free 掉的内存并不是都会马上归还给系统, 相反, 它建立了一系列内存缓冲区来统一管理内存块。

在大多数情况下，当用户进行下一次分配请求时, ptmalloc会优先在这些缓冲区中挑选空闲内存, 而不是直接通过系统调用向内核取获取内存；而当用户进行一次释放操作时，ptmalloc也会将内存块归还到缓冲区，仅当满足特定条件时，才会将批量的内存块一起归还给系统。这样的缓冲机制，不用每次分配和释放都向内核去获取和归还内存，避免了反复的系统调用, 能够显著减少程序频繁分配/释放内存的时间开销。

我们先来了解一下glibc库的ptmalloc机制的主要特点，然后再继续分析产品线的问题。

在ptmalloc分配机制中，空闲内存缓冲区主要包括top chunk，Fastbins和Bins三种。应用程序通过malloc申请内存时，ptmalloc总是以Fastbins->Bins->top chunk的顺序，依次在这3种缓冲区中尝试分配空闲内存。下面我们分别介绍这3种缓冲区的特点：

1．top chunk：

  程序整个HEAP（堆区）的顶部内存块，它是ptmalloc最先建立的缓冲区，在程序启动时就会为其分配好的一块较大尺寸的空闲内存。

  程序的第一次malloc操作所获得的内存块，就是从top chunk低地址端切割出来的；如果top chunk的的内存不够用了，就通过sbrk系统调用进行扩展；当程序将通过free释放内存块后，如果它在地址上紧邻top chunk，且没有被放入下面会介绍的Fastbins缓冲区，ptmalloc也会将它立即合并回top chunk中。

  对于程序后续的malloc操作，如果无法从Fastbins和Bins缓冲区分配到合适的内存块，最终也会再次从top chunk切割分配。

  在程序执行过程中，top chunk可能会多次通过sbrk来扩展。另一方面，如果某次free释放后发生了top chunk合并导致top chunk的尺寸达到某个门限值，则ptmalloc会将top chunk的高地址端的一部分通过munmap归还给系统，保证top chunk的尺寸在一定范围内。top chunk在多次分配和释放过程中发生的地址和尺寸变化 ，会直接引起程序物理内存消耗的增多或减少。

2．Fastbins：

  快速内存块链表，它是针对很常见的“反复分配/释放小块内存”的内存应用模式而建立的。当free的内存小于max_fast (默认值72 bytes)，这块内存就被放入Fastbins链表；如果下一次malloc申请的内存小于max_fast时，会优先从Fastbins链表中查找合适尺寸的小内存块提供给程序；于是，当程序反复分配/释放小块内存时，ptmalloc不需要执行复杂的分配逻辑，只需要在Fastbins链表中来回操作即可，大大提升了分配和释放效率。

  当某次malloc分配无法从Fastbins链表中找到合适的内存块时，ptmalloc就认为程序的“反复分配/释放小块内存”的过程已经结束，此时会销毁Fastbins链表，同时链表中挂接的小内存块会被转换为普通内存块Bins。

3．Bins：

  普通内存块链表，它们最初都是从top chunk切割出来的，但当它们被释放时，由于此时的top chunk又经历了若干次分配切割，地址已经变化了，所以这些被释放的空闲块在地址上与top chunk往往不再相邻，无法被合并，同时，尺寸也不满足Fastbins链表的要求，于是就被挂接到普通内存块Bins链表中。后续的malloc分配如果不适用Fastbins，则通过Bins链表从这些普通内存块中挑选合适的内存块。

  当一个普通空闲内存块被挂接到Bins链表时，ptmalloc还会尝试将它和与其地址相邻的空闲块进行合并，形成单个更大的空闲内存块，简化缓冲区空闲内存组织，利于提高后续分配的效率。

了解了这几个缓冲区的概念和特点后，我们再回头来看产品线这种动态内存应用方式，在glibc ptmalloc的层面发生了什么：

  完成内存释放后，ptmalloc动态内存布局与之前分配完成时完全一样，没有任何空闲内存块发生合并，所有空闲块都只能以bin的方式缓存于HEAP中；top chunk尺寸也没有通过空闲块合并获得增长，因此无法归还任何物理内存给系统。

综上所述，我们可以看到问题的全貌：

glibc库ptmalloc的内存缓冲机制有意利用物理内存归还滞后性来获得更快的内存分配释放执行速度，是一种典型的“内存-性能权衡”的方法。

碰巧的是，产品线程序在初始化阶段的内存分配与释放的模式，刚好将进程HEAP的动态内存置于一个缓存了大量空闲内存的暂态上。

其实，此刻如果程序再执行一次size大于Fastbins尺寸门限的malloc分配操作，就可以触发ptmalloc销毁Fastbin链表，随后合并大量的空闲块，最后通过top chunk将这些空闲内存都归还给系统，可是该程序在后续执行过程中恰恰又没有这样的操作，以至于程序在完成初始化过程后，并没有实际归还任何物理内存，它的物理内存占用始终保持在一个峰值水平上，远大于它们在初始化后持续运行的实际内存需求。

于是，当产品线的版本测试环境中启动了多个这样的程序实例后，每个实例都占用了过多的物理内存，最终造成了预料之外的系统物理内存耗尽。

解决方案

根据前节分析，产品线的程序从ptmalloc缓存特点来看，并不是一种合理的动态内存应用模式，我们可以针对ptmalloc缓存特点提出几种调整的思路：

1）改变内存分配方法，比如使用calloc（N,small_size）来替代N次malloc（small_size）的小内存分配，这样就不会有FastBin的介入，所有内存free后，都可以与top chunk合并，这样top chunk就可以在循环释放中，多次归还物理内存给系统；

2）改变分配顺序，比如先分配完所有小内存（位于Heap低地址端）再分配所有大内存，这样当所有内存free之后，虽然存在FastBin链表，但所有大内存释放后都会被合并到top chunk并陆续被归还给系统，最终整个HEAP浪费的内存最多只有N*small_size，也可以显著改善问题。

3） 此外，ptmalloc考虑到实际应用中可能出现本文案例的情况，也专门提供了一个malloc_trim()接口给应用程序调用，malloc_trim()接口会首先强制销毁HEAP中的FastBin链表，然后合并所有可合并的空闲块，最后尝试将top chunk高地址端内存尽可能地释放给系统。程序可以结合自身流程，在适当的时候调用malloc_trim()接口来强制归还物理内存。

产品线最终选择了第一种方法，解决了这一问题。

应用建议

 为避免频繁进行系统调用而带来性能的损耗，glibc堆内存的释放都是滞后于应用的free操作的。因此，用户在使用堆内存时，应避免出现不合理的动态内存使用模型。

除了根据前文介绍的glibc ptmalloc的缓冲机制特点来控制程序内存分配模式之外，在实际开发中，还可以通过glibc提供的mallinfo函数，对程序执行过程中的HEAP动态内存的使用情况进行查看，以便发现不合理的内存应用的迹象。

该函数的介绍如下：

函数原型：struct mallinfo mallinfo  ((void));

它将返回一个mallinfo的数据结构，该数据结构主要的成员及含义如下：

  • int arena：

    arena的值表示当前整个HEAP空间从系统中分配的内存大小，也即Bins + Fastbins + top chunk三种缓冲区此时的总大小，包含了所有空闲和占用的内存块；

  • int ordblks

    Bins和top chunk缓冲区空闲块的总数量（top chunk始终只有1个，所以ordblks-1 就是Bins缓冲区中的空闲块总数量）

  • int smblks

    快速内存块链表Fastbins当前缓存的小尺寸空闲块数量 

  • int fsmblks

    快速内存块链表Fastbins当前缓存的总空间；

  • int fordblks

    当前整个HEAP空间中缓存空闲内存的总大小；

  • int keepcost

    top chunk当前的大小；

int usmblks：整个HEAP空间在本进程的历史中分配的最大内存空间，也即HEAP尺寸峰值；如果曾经发生过使用将内存归还给系统的行为，该值可能大于arena；

  1）我们预期HEAP空间得当前尺寸arena应该已经收缩到循环分配之前的尺寸了，所以它应该明显小于usmblks记录的HEAP空间的峰值尺寸，而实际上却观察到arena与usmblks值很接近，说明当前HEAP空间并没有因为循环释放而发生明显的收缩。
