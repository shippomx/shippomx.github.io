业务容器内下载文件速率较慢的问题分析

哪吒藕霸2023-06-022023-10-08

二、问题分析

1.初步现象

业务同事发现通过curl -o test.apk -X GET https://a.com/test.apk 下载文件到容器内，节点上下载速100M+，但是业务容器内速率只有800k 左右。

ens3网口和cnio网桥分别进行抓包：

通过抓包的数据发现，以seq=17514 这个报文为例，在cnio网桥上的信息显示seq=17514之前的报文，都在收到seq=17514之后才陆续收到，明显出现了报文乱序的情况，同时也存在丢包。

2.确认丢包情况

初步确认了问题的现象后，联系业务同事进一步了解当前环境的网络链路结构，绘制如下示意图：

报文流向为：容器eth0(192.169.15.34)  –>  cni0 (192.169.15.1)    –>节点网卡 ens3 (10.54.147.178)   –> server (10.54.154.121)，其中出现丢包的位置是在cni0处。梳理了具体的网络链路后，能够直观的观察到报文是在虚拟网桥cni0上收包或者向容器veth口转发的时候存在异常。

接下来再次复现进行抓包确认，抓包信息如下图，从报文信息中发现长度太大的报文都存在丢失情况，初步怀疑可能和报文长度有关系，丢包导致大量乱序报文影响下载速度。按照常规手段复现时观察网卡统计项(ifconfig cni0)和内核统计项(/proc/net/snmp)，发现网桥cni0处TX dropped计数存在增加，进一步确定应该是转发过程中存在丢包，但仍不能确定丢包位置，接下来准备使用systemtap工具来进行确认。

3.寻找丢包位置

目前虽然已经清楚dropped计数有所增加，但仍然不清楚具体丢包的原因，因此需要寻找报文丢包的具体位置。当前环境复现操作方便(即容器内直接执行下载命令)，同时不是部署商用业务的环境，因此决定直接部署systemtap（内核引入的一种Kprobe机制，利用Kprobe 提供的API来实现动态地监控和跟踪运行中的Linux内核，可以用来动态地收集调试和性能信息的工具，使用时需要安装对应kernel版本的debuginfo包）工具进行确认。

首先是确认报文kfree_skb的情况，因为知道源端，目的端的ip地址以及对应dev设备名称，因此可以直接过滤抓取丢包的报文以及对应的堆栈信息，复现后发现是validate_xmit_skb函数中存在丢包。该函数中有三处goto out_kfree_skb，即都有可能是丢包的位置。根据走读代码发现segs = skb_gso_segment位置的丢包嫌疑最大，然后调整systemtap的脚本，通过行号定位确认是哪一个goto流程。

最终打点验证后，发现的确是skb_gso_segment返回异常导致的丢包，因为通过3332行号能够过滤对应的打印信息。目前根据已有的信息，初步怀疑与网卡的gso或tso配置相关，导致报文传输过程中延时分片，最终造成了报文丢包。本以为对环境网卡上的gso和tso关闭后，故障不会出现，但复现过程中发现设备cni0上tx dropped的统计仍然增加，而且下载速度和之前一样。决定还是得刨根问底，查一查异常的根因才能让真相浮出水面。

4.排查丢包根因

根据前面的分析，已经知道了丢包的位置，目前只需找到skb_gso_segment函数中的异常的返回位置即可。查看该函数代码后，需一层一层调用查看return返回置。后续发现了其返回值segs = ptype->callbacks.gso_segment(skb, features)的可疑位置，由于报文地址是ipv4，协议是tcp，通过打点函数的返回值确认了是tcp4_gso_segment函数中返回值了异常值-22，原因是skb中gso_type标志不是SKB_GSO_TCPV4。

关于这个type标记为什么不是SKB_GSO_TCPV4，是我们下一步需确认的问题。目前需关注什么时候会赋值gso_type，这样就能顺腾摸瓜，找到异常的位置，真相离我们已经不远了。接下来就继续走读代码，找到组装skb结构体，即gso_type赋值的位置。前面我们已经熟悉了网络结构，报文流向是从后端进ens3口，通过ip转发到cni0网桥，所以去找virtio驱动后端组装skb的位置，从代码中找到了receive_buf函数，搜索gso_type关键词，其中发现gso_type的赋值操作是在virtio_net_hdr_to_skb函数中，其异常标记是通过判断hdr->hdr.gso_type的标记并其标记赋给skb，代码如下：

其中hdr对应 hdr = skb_vnet_hdr(skb) ，对receive_buf打点发现skb的赋值是在skb = receive_mergeable这个流程里，并进一步确认了skb是从receive_queue获取的。

梳理了代码流程后再次打点，新增打印hdr->gso_type的标记以及napi_gro_receive函数中gso_type的标记，复现后过滤异常报文的skb地址:0xffff99fee66a0d00，发现报文在ens3网卡就携带了异常gso_type，同时hdr_type为VIRTIO_NET_HDR_GSO_TCPV6，导致后续tcp4_gso_segmentz中gso_type不匹配，而报文head_skb是从后端receive_queue中得到的。

目前真相已浮出水面，找到了异常的gso_type标记，并确认type 18的标记是通过virtio_net_hdr_to_skb赋给的，将virtio_net_hdr 中gso_type赋给了skb，但struct virtio_net_hdr *hdr是receive_queue传入的，需确认是否dvs后端有过什么修改配置，导致报文进来就携带了标签，造成后续不匹配SKB_GSO_TCPV4出现丢包，影响下载速度。

三、问题结论&解决方案

1.问题结论

通过上述一系列排查，后端发送的报文中gso_type携带了异常标记，在cni0网桥进行转发时tcp4_gso_segment函数中返回了错误码-22，导致validate_xmit_skb处出现丢包。主要原因与dvs的gro/gso功能有关，其中2020年滨江环境gro/gso功能不完善导致业务性能低，后续dvs版本优化了该部分功能。

2.解决方案

建议升级dvs版本，当前dvs版本较低，建议升级版本03.19.30.P7版本，或者对齐中移现场升级的11版本。后续dvs升级后容器内下载文件，没有出现丢包的情况，下载速度恢复正常。

四、问题总结

1.排查思路

问题分析整体思路：

  • 先确定网络拓扑结构或链路情况，这样能够直观清晰的了解问题背景，方便排查和定位。

  • 对问题复现抓包，确认丢包位置，一方面也能定界问题。

  • 如确认是系统侧丢包，查看相关统计项快速定位丢包位置。

  • 如统计项显示不太明显，可使用systemtap等工具进行确认。

  • 结合代码进行分析丢包的原因。

2.问题小结

在处理外场等多种问题中，经常会碰到丢包的网络故障，通常情况下是查看网卡以及内核相关统计项来判断，根据统计项计数的变化可以确定是否存在缓存区溢出，checksum异常或数据包自身错误等，但很多时候由于其他原因的丢包，统计项并没有完全统计到，因此仅仅依靠统计项判断是完全不够的，常常还需要通过dropwatch、systemtap等工具来协助进行确认和排查。本文通过一个外场故障实例对网络中遇到丢包的情况进行了分析和总结，该案例具有一定的代表性，从中我们看到网卡上drop统计计数项虽然有增加，但不是常规的buffer溢出，因此丢包位置还是较为隐蔽，需要多次打点进行复现排查。如果单独对atomic_long_inc(&dev->tx_dropped)进行过滤，存在丢包的可能性很多，因此最好是打点kfree_skb来进行判断。通过systemtap打印信息提供的线索，结合代码流程一步一步梳理，最终找到丢包位置并确认丢包的原因。

在处理网络丢包问题中，首先应该对网卡设备、网络协议栈有初步的认识，了解不同的协议类型，报文的收发流程；熟悉systemtap工具，包括一些基础使用命令和嵌c的用法，利用调试工具对报文过滤从而准确定位丢包位置以及打印所需的一些结构变量等；在处理钩子函数调用的时候，即ptype->callbacks.gso_segment，能够根据代码或者ftrace等手段来确认判断。

本次案例中的感悟：在处理故障时可通过对比正常环境，寻找其差异性，可能会更快速的定位问题。在systemtap打点过程中，不要怕麻烦，有些流程不是很容易判断，需要多次打点行号或者返回值等才能确认流程，同时需熟悉代码及网络相关流程，这样在定位问题和调整systemtap脚本方面会少走弯路。
